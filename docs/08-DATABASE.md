# Database — Supabase Postgres + pgvector

Full DDL lives in `backend/migrations/001_initial_schema.sql` — this doc explains the *why* behind each table/policy/function, which the SQL file itself only lightly comments. The migration file is a **consolidated snapshot reconstructed from the live schema** (via `information_schema`/`pg_catalog` introspection), not a literal replay of every incremental change — most of this schema was built directly against the running Supabase project across many sessions and had never been committed to the repo before that reconstruction.

## Tables

**`documents`** — one row per uploaded file. `chunk_count` is overloaded as a status field: `-1` means "still processing" (the background-ingestion sentinel, doc 2), any non-negative integer is the real, final chunk count. There is no separate `status` enum column — this was a deliberate minimal-schema-change choice when background ingestion was introduced, reusing a column that already existed rather than adding a migration for a new column. `folder` is nullable free text (no separate folders table — folders are just a label on the document, not a first-class entity with their own metadata).

**`chunks`** — the vector store. `embedding vector(384)` (dimension is a hard schema constraint — the embedding model can be swapped for another 384-dim model with no migration, but swapping to a different-dimension model *does* require an `ALTER TABLE` in addition to re-embedding everything). `content_tsv tsvector generated always as (to_tsvector('english', content)) stored` is a **generated column** — Postgres maintains it automatically on every insert/update to `content`, so the keyword-search half of hybrid retrieval never has to remember to keep a derived column in sync. `page_number` defaults to `0` for loaders that don't have a native page concept (TXT, MD, DOCX — only `PyPDFLoader` sets a real page number). The `gin` index on `content_tsv` is what makes `keyword_search_chunks`'s `websearch_to_tsquery` lookups fast; **there is deliberately no vector index (HNSW/IVFFlat) on `embedding` yet** — at current data volume a sequential scan is fine, and this is explicitly flagged in the main `README.md`'s "not done" section as the next real scaling concern, not an oversight.

**`sessions`** — raw turn-by-turn chat message history (one row per message, not per session). Named `sessions` rather than `messages` for historical reasons — `chat_sessions` (below) is the newer table that actually represents a "session" as a first-class entity; the naming is a real, mildly confusing artifact of the product growing multi-session support after the fact, worth knowing so you don't assume `sessions` holds one row per conversation.

**`chat_sessions`** — session *metadata*: display name, creation/last-active timestamps, and the sharing fields (`share_token`, `is_public`). `id` is `text`, not `uuid` — sessions are created client-side with a generated ID before the first message is ever sent, and the app generates them as UUID-format strings but the column itself doesn't enforce the `uuid` type, so nothing stops a non-UUID string from being written here if a future code path ever changes ID generation.

**`graph_nodes`** / **`graph_edges`** — the knowledge graph. `graph_nodes.node_id` is `unique` (globally, not per-user — meaning if a `unique` constraint check is ever accidentally queried without a `user_id` filter, it will collide across users' distinct concepts that happen to share the same lowercased text, e.g. two different users both having a "process" node). In practice every write and read in `graph/store.py` filters by `user_id` explicitly, so this is a latent schema sharpness rather than an active bug, but it's the one place in the schema where the uniqueness constraint doesn't itself carry the tenant-isolation guarantee — RLS and application-level filtering are doing that work instead.

**`query_cache`** — backend-only. No client (browser) ever reads or writes this table directly; it exists purely for `rag/cache.py`'s server-side cache. RLS is enabled with **zero policies**, which is intentional: an RLS-enabled table with no policies denies all access by default for any role subject to RLS — since the backend's Supabase client uses the service-role key (bypasses RLS entirely), this table is effectively backend-only by construction, not just by convention.

**`rate_limits`** — same "backend-only, RLS-enabled-with-no-policies" pattern as `query_cache`. Doubles as storage for two logically separate things (request-rate windows via `check_rate_limit`, and daily token-usage windows via `increment_token_usage`) that share the exact same windowed-counter shape and just use different key prefixes (`{action}:user:{id}` vs `tokens:user:{id}`) — one table, one RPC pattern, two call sites, deliberately not split into two tables since the underlying mechanism is identical.

**`user_preferences`** — one row per user (`user_id` is the primary key, not a separate `id` — a user can only ever have exactly one preferences row, enforced by the schema itself rather than application logic). `priorities` is stored as a comma-joined `text`, not a Postgres array or a join table — a deliberate simplicity choice for a small, low-cardinality, order-preserving list that's always read/written as a whole, never queried by individual element.

**`user_memory_notes`** — one row per manually-added memory note. No auto-extraction pipeline writes here; every row exists because a user explicitly typed it (see doc 3's note on `memory.py`).

## Row Level Security

Every user-facing table has one policy: `for all using (auth.uid() = user_id)` (a couple also add `with check (auth.uid() = user_id)` for the two tables where a user can *insert* rows directly tied to themselves — `user_preferences`, `user_memory_notes` — the `with check` clause additionally prevents writing a row with someone else's `user_id`, which matters more for insert/update than the read-side `using` clause alone covers).

**This is defense-in-depth, not the primary access control.** The backend's Supabase client always uses the service-role key, which bypasses RLS entirely — every actual tenant-isolation guarantee in this system comes from application code explicitly filtering every query by `user_id`, itself derived from a verified JWT (`get_current_user` in `app.py`). RLS exists so that *if* a future code path ever used the anon/authenticated key directly from the browser (which nothing currently does — the browser only ever talks to the backend, except for Supabase Auth itself), it would still be safely scoped. It's a safety net for a code path that doesn't exist yet, not the thing currently doing the work.

## RPC functions

**`match_chunks(query_embedding, match_count, similarity_threshold, p_user_id, p_document_ids)`** — pgvector cosine similarity (`1 - (embedding <=> query_embedding)`), filtered by user and optionally a specific set of document IDs (the "scope this question to specific documents" feature), thresholded to drop low-similarity noise before it ever reaches fusion. Note there are (or were, at time of schema introspection) **three overloaded versions** of `match_chunks` live in the actual Supabase project — a 2-argument version, a 3-argument version with just a threshold, and this full 5-argument version — left behind from the feature's evolution. Only the 5-argument version is called anywhere in the current backend code; the migration file in this repo only defines that one. If you're working directly against the live Supabase project (not a fresh one built from the migration file), be aware the older overloads may still exist there as harmless-but-confusing dead functions.

**`keyword_search_chunks(query_text, match_count, p_user_id, p_document_ids)`** — Postgres full-text search via `websearch_to_tsquery` (handles natural "quoted phrases" and `-exclusions` the way a search-engine query box would, more forgiving than plain `to_tsquery`) against the generated `content_tsv` column, ranked by `ts_rank`.

**`check_rate_limit(p_key, p_limit, p_window_seconds)`** — the windowed-counter pattern: insert-or-increment a row in `rate_limits`, resetting the counter if the window has expired, returning whether the *post-increment* count is still within limit. Insert-then-check (not check-then-insert) — this matters for correctness under concurrent requests: two simultaneous requests both incrementing the same key will both see accurate post-increment counts rather than racing on a stale read.

**`increment_token_usage(p_key, p_tokens, p_window_seconds)`** — identical shape, accumulating token counts instead of request counts, returning the running total (not a boolean) since the caller needs the actual number to compute a percentage against `DAILY_TOKEN_BUDGET`.
