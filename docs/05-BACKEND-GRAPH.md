# `backend/graph/` — knowledge graph extraction and storage

Two files. This is genuinely the smallest, simplest module in the backend, and also the one with the clearest "this is a seed of something bigger" future-scope story (see the main `README.md`'s "Future scope" section).

## `extractor.py` (80 lines, the whole file)

One function, `extract_entities_and_relations(text, source)`. One Groq call (`llama-3.3-70b-versatile`, the same primary model as the main answer path, via its own cached singleton — not shared with `retrieve1.py`'s `get_llm()`, a separate `_EXTRACTOR_LLM` module-level cache with the identical multi-key-fallback pattern). The prompt asks for strict JSON: up to 10 lowercase entities, up to 6 subject/relation/object triples, "only extract explicitly stated relationships" (an instruction to suppress the model inferring relationships that aren't actually in the text — a real hallucination-suppression concern for a task that's otherwise unconstrained by retrieval/grounding the way the main answer path is).

Input is truncated to the first 800 characters of the chunk (`text[:800]`) — extraction runs per-chunk (up to 10 chunks per uploaded document, see doc 2's `_build_graph_for_chunks`), so this bounds the per-call cost; a chunk's *later* content beyond 800 characters simply doesn't contribute entities/relationships, which is a real, accepted quality ceiling, not a bug.

Fails soft: a JSON parse failure or any other exception returns an empty result (`{"entities": [], "relationships": [], "source": source}`) rather than raising — one chunk's extraction failing during a background ingestion job should never abort the graph build for the other 9 chunks (`_build_graph_for_chunks` in `app.py` already wraps each chunk's call in its own try/except, so this internal fail-soft is a second layer of the same protection).

## `store.py` (168 lines)

`add_to_graph(extracted, user_id)` — the write path. For each entity: check if a `graph_nodes` row already exists for `{node_id, user_id}` (node IDs are lowercased/stripped in the extractor's own prompt instructions, so matching is effectively case-insensitive by construction); if it exists, append the new source filename to its `sources` array (deduplicated — `if source not in current_sources`); if not, insert a new row. For each relationship: ensure both the subject and object exist as nodes (inserting bare nodes with just the source if they don't — a relationship can introduce an entity that wasn't independently extracted as a standalone entity), then insert an edge if the exact `{source, target, user_id}` triple doesn't already exist (relation *text* isn't part of the uniqueness check — two different relation descriptions between the same pair of entities are still deduplicated to one edge, keeping whichever relation text was inserted first).

Note this write path is not batched — it's one Supabase call per entity-existence-check, one per entity-insert-or-update, one per edge-existence-check, one per edge-insert, all sequential. For up to 10 entities and 6 relationships per chunk, across up to 10 chunks per document, this is a real number of small round trips per upload (bounded, but not optimized) — acceptable because it all runs in a background task that doesn't block the user-visible response, but worth knowing if graph-building latency ever becomes a concern.

**Read path caching** (`_get_graph_data`, `get_full_graph`, `get_related_nodes`): both read functions used to re-scan the *entire* `graph_nodes` and `graph_edges` tables from Supabase on every single call — opening the graph panel and clicking a topic both paid a full-table-read cost even when nothing had changed since the last read. Fixed with an in-process, per-user, 60-second TTL cache (`_graph_cache`, same pattern as `rag/cache.py`'s `has_any_chunks`) behind a shared `_get_graph_data(user_id)` helper both read functions now call. `add_to_graph` invalidates the cache (`invalidate_graph_cache(user_id)`) once at the end of every write, so a read is never more than one write-cycle stale.

`get_related_nodes(topic, user_id, depth=2)` does the actual BFS: find nodes matching the topic (exact match first, then substring match either direction, then a word-overlap fallback for multi-word topics — three escalating match strategies, each only tried if the previous one found nothing), then walk up to `depth` hops outward along edges, collecting every node/edge touched along the way. This is what powers "related concepts" under every chat answer and what the frontend's `GraphPanel` renders when a user clicks into a topic.

`get_full_graph(user_id)` is the same underlying cached data, reshaped as the complete graph with no BFS filtering — what powers the "view full graph" button.
