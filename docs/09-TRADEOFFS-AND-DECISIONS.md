# Tradeoffs and decisions

The main `README.md` has a condensed version of this table aimed at a reader deciding whether to hire someone. This is the fuller version, including decisions made *within* the RAG pipeline itself, not just the top-level stack choices.

## Infrastructure-level (see main README for the full rationale on each)

| Decision | Chosen | Rejected | Core reason |
|---|---|---|---|
| Vector store | Supabase pgvector | FAISS | Render's filesystem is ephemeral |
| Embeddings (runtime) | fastembed, local | `sentence-transformers` | `torch` OOM'd the 512MB box on startup |
| Embeddings (API alternative) | — | Hugging Face Inference API | Render blocks outbound DNS to `api-inference.huggingface.co` |
| LLM | Groq | Self-hosted Ollama | Render has no GPU |
| Reranker | FlashRank (ONNX, local) | Cohere Rerank API | No extra API key/network hop/cost, at some ceiling-accuracy cost |
| Result fusion | Reciprocal Rank Fusion | Score concatenation / take-top-k-per-method | Sidesteps normalizing two incomparable score scales |
| Eval methodology | In-process pipeline call | Live API over HTTP | Original HTTP approach was a workaround for a since-properly-resolved dependency conflict |

## Retrieval-pipeline-level

**Confidence-gated generality (Corrective RAG) over a binary refuse/answer split.** The alternative to CRAG is simpler: either strictly refuse anything not well-supported by retrieval, or always answer from general knowledge when unsure. Strict refusal throws away the model's actual reasoning ability whenever retrieval is merely *imperfect* rather than *absent* — a real cost for a system meant to be genuinely useful, not just safe. The chosen design (confidence threshold gates which system prompt applies, and the general-knowledge path is required to self-label as such) keeps both the safety property (never claim something is from your documents when it isn't) and the usefulness property (still answer when it reasonably can), at the cost of a tunable threshold that has to be calibrated against real eval data rather than picked once and forgotten — this project's threshold moved from `0.35` to `0.45` specifically because eval data showed weakly-relevant retrievals passing the original bar and dragging down faithfulness.

**Conditional Self-RAG (skip above a confidence bar) over unconditional verification.** Verifying every single grounded answer is the "obviously correct" default and was the original design — but it's a second sequential Groq call on every answer, real latency for the common case. The empirical bet made here: retrieval confidence already correlates with how likely an answer is to be well-supported, so trusting the STRICT RULE prompting alone above a bar (`0.75`) recovers that latency for most answers while keeping the safety net exactly where it's needed most — low-confidence, borderline retrievals, which is also where the STRICT RULE prompting alone is least likely to be sufficient on its own. This is a bet, not a proof — it hasn't been eval'd specifically for whether skipped-verification answers are in fact reliably fine at high confidence vs. just usually fine. Worth a dedicated eval pass if accuracy regressions ever show up specifically in the high-confidence bucket.

**Conditional query rewriting (reference words OR short questions) over always-rewrite or never-rewrite.** Always-rewriting adds a Groq call to every query for no benefit on already-explicit questions. Never-rewriting leaves genuinely ambiguous follow-ups ("what about that?") and under-specified short questions ("deadlock?") to retrieve poorly with no recourse. The chosen middle ground uses two independent cheap signals (a reference-word keyword match, or a word-count threshold) to decide, rather than an LLM call to decide whether an LLM call is needed (which would defeat the purpose). The `<=4 words` threshold is a heuristic, picked by judgment rather than tuned against eval data specifically for that boundary — a reasonable next experiment if retrieval quality on short queries is ever specifically in question.

**Reasoned hyperparameter defaults over exhaustive grid search.** `k` (5→8→6), chunk size (500/50→1000/150), and the CRAG threshold (0.35→0.45) were all changed together in one pass, validated with one before/after RAGAS comparison, not independently swept across a grid of values with isolated before/after runs per parameter. The honest reason is resource-constrained, not methodological: a single 8-question eval run against Groq's free tier already takes 8-11 minutes with heavy `429` retries (25 questions takes proportionally longer — see the eval-results section of the main README for actual measured runtimes). A true grid search — even just 3 values × 3 parameters, isolated — would be 9+ full eval runs, each burning real Groq quota that the app's own daily-budget mechanism (`token_usage.py`) is explicitly designed to protect. The one exception: `k` specifically *was* independently tested three times (5, 8, 6) because it was cheap to vary without re-ingesting or re-embedding anything, unlike chunk size (requires re-ingestion) or the embedding model (requires a full re-embed migration).

**Embedding model swap validated via before/after eval, not a documented ablation isolating just the embedding change.** The `k=8` eval run changed the embedding model, chunk size, and CRAG threshold simultaneously (see above) — so the measured recall/precision shift can't be cleanly attributed to the embedding swap alone versus the chunking change alone. If this project needs to defend "the embedding swap specifically was worth the re-embedding migration cost," that requires a dedicated isolated comparison that hasn't been run.

## Architectural-level

**Everything in one FastAPI process, no queue.** Background work (ingestion, graph extraction) runs via `BackgroundTasks` in-process, not a separate worker/queue (Celery, RQ, etc.). This is appropriate at current scale and given Render's free tier is a single process anyway — introducing a queue would mean either paying for a second Render service (defeating the point) or running a queue *within* the same single process, which buys nothing over `BackgroundTasks` directly. This stops being the right call the moment concurrent ingestion load becomes real (multiple users uploading large documents simultaneously would all compete for the same `ThreadPoolExecutor(max_workers=4)` and the same weak single vCPU) — worth revisiting if/when this project has to handle that.

**One massive frontend file over component-per-file.** Discussed in doc 7. Named here again because it's a real, load-bearing tradeoff for anyone taking over: the cost isn't runtime (React doesn't care how many files a component is defined across), it's onboarding and merge-conflict surface area for a team larger than one.
